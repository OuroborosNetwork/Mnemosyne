## 1. Vision

### 1.1 The two problems being solved

**Problem 1 — Origin fragmentation.** The OuronetUI codex today lives in browser `localStorage`, tied to the page's origin. When OuronetUI migrates from `devwallet.ouronetwork.io` to `wallet.ouro.network`, each user's codex stays trapped under the old origin. Same when AncientHoldings's admin UI moves to `codex.ancientholdings.eu`. Users would have divergent codices per origin with no automatic way to unify them. The same problem appears when a user wants to use OuronetUI on both desktop and mobile — separate localStorage means separate codices.

**Problem 2 — Transparent codex provisioning for non-crypto users.** Future consumer services in the StoaOuronet ecosystem (most concretely: an upcoming movie-streaming platform) will let users register with `{email, password}`. The vision is that those users *transparently* get an ouronet address as a side effect — they don't need to know what a codex is, what a mnemonic is, or what self-custody means. They register for movies; the system happens to have minted them a wallet they can use elsewhere later (e.g. on OuronetUI, with the same credentials, their codex is already there). When ready to learn, they can "graduate" to self-custody.

### 1.2 What Mnemosyne is

A **centralized SaaS** that stores user codices encrypted-at-rest, serves them to consumer services (OuronetUI, AncientHoldings hub, the movie platform, anything else) under a unified credential, and enables transparent codex provisioning for non-crypto users. Architecture is patterned after **Bitwarden** — well-proven, well-understood, shippable in weeks rather than years.

### 1.3 What Mnemosyne is NOT

- **NOT a decentralized protocol.** Earlier exploration considered building a Bitcoin-like protocol with cryptographically-decentralized governance. That's a multi-year research project; out of scope. Decentralization happens in layers via optional Arweave snapshots and on-chain Stoa pointers (see §8) — but the v0.1 core is a centralized service.
- **NOT a custodial wallet service in the regulatory sense.** The cryptographic design makes Mnemosyne **technically non-custodial**: the operator cannot decrypt user codices for self-custody users. For grandma-tier users, the operator can theoretically assist with recovery via wrapped recovery factors — clearly documented opt-in.
- **NOT a replacement for the `@stoachain/ouronet-codex` package.** It's a new *adapter* for the existing package — `MnemosyneCloudAdapter` — alongside the existing `LocalStorageCodexAdapter` and `MemoryCodexAdapter`. The codex data model, signing pipeline, and all existing surfaces are unchanged.
- **NOT an identity provider for non-codex purposes** (yet). Initial scope: codex storage + retrieval. SSO-style "Sign in with Mnemosyne" is a future possibility, not a launch goal.
- **NOT a regulated entity** at v0.1. Operator (Ancient Holdings) intends to convert to a foundation later; the technical design is jurisdictionless-by-construction (no operator privileged access to user keys), but the *operator* still has to think about its legal posture for the SaaS layer. See §12.

### 2.2 Why split static frontend / dynamic backend onto different domains

- **Static frontend can be Arweave-permaweb-deployed.** Vite produces a `dist/` folder; that bundle posts to Arweave one-time and stays online forever. Even if Cloudflare drops the `mnemosyne.ancientholdings.eu` domain, users can hit `arweave.net/<tx-id>` and the SPA still works (it just talks to the same backend API).
- **Backend cannot be permaweb** — it has a database, session state, dynamic queries. Must be a regular server.
- This is a common pattern for "decentralization-grade" web apps that need some centralized state. OuronetUI itself should follow the same pattern eventually (its SPA can be permaweb-deployed; the Stoa nodes it talks to are already decentralized).

| Layer | Choice | Rationale |
|---|---|---|

| Backend framework | TBD (Next.js / Express / Hono) | Open question — see §10 |
| Database | TBD (Postgres / SQLite) | Open question — see §10 |
| Auth | TBD (roll-our-own Argon2id / Lucia / Auth.js / Supabase) | Open question — see §10 |
| Password hash | Argon2id (parameters tuned for ~500ms) | Industry standard for password hashing |
| Codex blob encryption | AES-256-GCM | Authenticated encryption; standard |
| Key derivation (password→key) | Argon2id-KDF or HKDF over Argon2id-hash | Standard |
| Signing primitive (recovery) | Schnorr signatures on Dalos curve | Already exists in `@stoachain/stoa-core/signing` |
| Hosting | Hetzner VPS (same as AH hub) | Owner's existing infra |
| Domain | `mnemosyne.ancientholdings.eu` + `api.` subdomain | Settles the consumer integration story |

## 3. Storage model

### 3.1 What the server stores per user

```
users table:
  id                           (UUID)
  email                        (unique, indexed)
  password_verifier            (Argon2id hash of password, with salt)
  codex_prime_address          (ouro address of CodexPrime, for on-chain pubkey lookup)
  guardian_addresses[]         (additional ouro accounts designated as Guardians)
  created_at, last_login_at

codex_blobs table:
  user_id                      (FK)
  encrypted_codex              (AES-256-GCM ciphertext of the codex JSON)
  encrypted_codex_key_password (codex encryption key wrapped by password-derived key)
  encrypted_codex_key_facts    (codex encryption key wrapped by facts-derived key) [grandma tier only]
  facts_hash                   (Argon2id hash of canonical(personal facts)) [grandma tier only]
  schema_version               (data schema version of the codex JSON inside)
  version                      (monotonic counter for optimistic concurrency)
  last_updated_at, last_updated_by_device

audit_log table:
  user_id, event_type, ip, user_agent, timestamp
  events: signup, login, password_reset, recovery_attempt, codex_update, ...
```

**Key invariant:** the server stores ONLY ciphertexts and verifiers. It cannot decrypt any user codex with anything it holds. Decryption requires the user-side password (or recovery facts, for grandma tier).

### 3.2 What flows over the wire

- On `GET /codex`: server returns `encrypted_codex` + `encrypted_codex_key_password`. Client decrypts locally.
- On `PUT /codex`: client uploads new `encrypted_codex` (re-encrypted under the same key). Server doesn't see plaintext.
- Server never sees the codex contents, the password, or the codex key in plaintext.

### 3.3 Encryption envelope (the "wrapping" model)

Codex contents are NOT directly encrypted with the password. Instead:

1. A **random Codex Key (CK)** is generated at signup (256-bit random).
2. The codex JSON is AES-256-GCM encrypted with CK → stored as `encrypted_codex`.
3. CK is then **wrapped** (encrypted) with a key derived from the user's password → stored as `encrypted_codex_key_password`.
4. For grandma tier, CK is *also* wrapped with a key derived from canonicalized personal facts → stored as `encrypted_codex_key_facts`.

Why this matters:
- **Password change preserves codex content.** When the user changes password, only `encrypted_codex_key_password` is re-computed. The much larger `encrypted_codex` blob is untouched, because CK didn't change. Same applies to all the contents inside (Address Book, watch list, settings, every seed/account) — preserved verbatim.
- **Recovery preserves codex content.** Same reason. Recovery via mnemonic or facts unwraps CK without changing it.
- **Multiple unwrap paths possible without re-encrypting the blob.** Useful for adding Guardians, multi-device keys, etc.

> Note: I had been using "CEK" (Codex Encryption Key) in earlier discussion. Renaming to **CK** (Codex Key) here for cleaner terminology — there's no real semantic difference but "CEK" was a Claude-coined term that doesn't need to stick.

### 3.4 The 3-layer storage architecture (locked 2026-05-28 Mnemosyne-Ledger pass)

The earlier §3.1-§3.3 describe what the Mnemosyne backend stores. The full storage picture is **three layers** with distinct contracts:

```
┌─────────────────────────────────────────────────────────────────────┐
│ LAYER 3 — Stoa chain (immutable trust anchor; always present)       │
│                                                                     │
│   Pact module: mnemosyne.CODEX                                      │
│   Tables:                                                           │
│     codex-identities    (one row per registered Codex Identity)    │
│       key:              apollo-address                              │
│       immutable:        codex-public-id, codex-guard,               │
│                         registered-at, registered-by                │
│       mutable:          arweave-tx-id, arweave-updated-at,          │
│                         arweave-backup-count                        │
│     codex-arweave-history (one row per backup; append-only)         │
│       key:              "<apollo-address>|<arweave-tx-id>"          │
│       fields:           codex-public-id, arweave-tx-id, uploaded-at │
│                                                                     │
│   Cost: tiny gas per pointer update (size-independent)              │
│   Trust anchor: PrimePureKey signs all mutations                    │
│   Censorship-resistant: any Mnemosyne-compatible client can read    │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 2 — Arweave (OPTIONAL, opt-in per user, paid via Bundlr/Irys) │
│                                                                     │
│   Per-codex encrypted blob, permanent storage                       │
│   Identified on L3 by arweave-tx-id (43-char base64url)             │
│   Cost: one-time AR payment per upload; no recurring                │
│   Encryption: same CK that wraps the L1 blob — no re-encryption     │
│   Use case: ultimate disaster recovery if Mnemosyne L1 dies         │
├─────────────────────────────────────────────────────────────────────┤
│ LAYER 1 — Mnemosyne PostgreSQL (PRIMARY hot path; always present)   │
│                                                                     │
│   Live encrypted blob, mutated on every codex change                │
│   Whole-blob encryption (Option A from design discussion)           │
│   API: GET/PUT /codex/{apollo-address}                              │
│   Cost: Mnemosyne operational                                       │
│   GDPR-deletable on user request (real "right to be forgotten")    │
└─────────────────────────────────────────────────────────────────────┘
```

#### 3.4.1 Why three layers, not one

- **L1 alone** = standard SaaS. Fast, cheap, but Mnemosyne dying = all data lost. Operator-controlled cache.
- **L1 + L2** = backup story. Mnemosyne dying = users with L2 opt-ins recover. But no way to *trust* a fetched L2 blob is the right one (anyone could publish a fake blob claiming to be yours).
- **L1 + L2 + L3** = full trust anchor. The chain entry tells you which Arweave tx-id is canonically yours. Mnemosyne dying = users with L2 opt-ins recover and can verify they got the right blob. The chain proves authenticity; Arweave provides the bytes; Mnemosyne provides hot-path UX.

The chain pointer is what makes Arweave a *backup* rather than just *another place data sits*. Without L3, an attacker could publish thousands of decoy blobs to Arweave and a recovering user would have no way to tell which one is theirs.

#### 3.4.2 What rejected the "whole-blob to chain directly" alternative

Initial brainstorming considered storing the encrypted codex blob *directly* on Stoa-chain as a third field on the Codex Identity entry. Rejected because:

| Issue | Whole-blob on chain | This 3-layer design |
|---|---|---|
| Stoa tx size limit (~75KB raw blob max) | Power-user codices don't fit | Arweave has no practical size limit; chain only stores 43-char tx-id |
| Gas cost on every codex mutation | Scales linearly with blob size; expensive for power users | Constant tiny gas (just pointer update) |
| HNDL exposure (harvest-now-decrypt-later) | Every blob permanently on public ledger; ~30-year horizon risk if AES-256-GCM ever breaks | Only opted-in users expose blobs; opt-out users keep blobs in deletable L1 only |
| GDPR deletability | Impossible (chain is immutable) | L1 supports real deletion; L2 is opt-in trade-off the user makes consciously |
| Permaweb-compatibility | N/A (chain is the storage) | Native (Arweave IS permaweb) |

The Mnemosyne owner's insight: separating storage layer (L2 Arweave) from trust anchor (L3 chain pointer) collapses every objection.

#### 3.4.3 What rejected "monthly auto-Arweave for everyone"

A variant proposal would have Mnemosyne periodically upload all user codices to Arweave automatically, regardless of opt-in. Rejected because:

1. **HNDL re-enters through the back door.** Users who didn't opt in get their blobs permanently public anyway. Contradicts the "most private cloud" pitch.
2. **Operational complexity.** Cron, retry, batch billing, size-limit edge cases, "what if Arweave is unreachable that month."
3. **Cost.** Even at $0.001/blob/month × thousands of users, Mnemosyne pays for a service most users didn't ask for.
4. **GDPR impossibility.** "Delete my Arweave copy" can't be promised; auto-upload would make this an unkeepable contract.
5. **Trust model leak.** Users wouldn't be able to verify per-upload whether the right ciphertext under the right key was actually sent.

Pure opt-in keeps the trust model honest, the cost model simple, and the privacy story consistent.

#### 3.4.4 Disaster recovery ladder

Worst-case ordering of how a user gets their codex back:

1. **Normal operation:** L1 is live, fast path. SPA fetches `GET /codex/{apollo}`, decrypts client-side.
2. **L1 down briefly:** SPA retries. Mnemosyne's uptime target should be 99.9%+.
3. **L1 lost permanently** (DB destroyed, operator shutdown, jurisdictional seizure):
   - User who opted into L2: SPA reads L3 chain entry → gets `arweave-tx-id` → fetches blob from `arweave.net/<tx-id>` or any gateway → decrypts. **Full recovery, even with zero Mnemosyne infrastructure.**
   - User who did NOT opt into L2: Has only their on-chain accounts. Codex address book, watch list, settings, herotag claim history all lost. Must re-create from on-chain account list. (Same outcome as if they'd never used Mnemosyne.)
4. **L2 lost** (catastrophic Arweave failure, never observed in practice): Same as case 3 for opt-in users — fall through to "rebuild from on-chain accounts."
5. **L3 lost** (mnemosyne.CODEX module corrupted on Stoa chain — would require a chain-wide rollback): Pact's immutability guarantees this doesn't happen in normal operation. If it did, the operator's recovery plan is a chain-level emergency response, not a per-user concern.

**Property to preserve:** the recovery from "L1 gone" works *without Mnemosyne existing at all*. That's the property that distinguishes "real backup" from "operator-controlled cache."

## 4. Cryptographic primitives & custody model

### 4.1 Two tiers, user picks at signup

| Tier | Recovery method | Cryptographic guarantee | Best for |
|---|---|---|---|
| **Self-custody** (default for crypto-natives) | Mnemonic only (proves ownership via Schnorr-sig vs on-chain CodexPrime pubkey) | Strong — admin cryptographically *cannot* help. Recovery requires same entropy as the original seed. | OuronetUI users, valuable codices, paranoid users |
| **Facts-recovery** ("grandma tier") | Personal facts + secret word (wraps CK; admin theoretically could brute-force with full DB access if facts are weak) | Medium — relies on attacker not having all the facts AND the DB AND a weak secret word. Mitigated by Argon2id. | Movie platform onboarding, low-value codices, casual users |

User can **graduate** from facts-recovery to self-custody at any time: "I'm ready, show me the mnemonic, let me write it down, remove the facts-recovery wrapper." Forward-path is built-in.

### 4.2 Self-custody recovery flow (the elegant one)

Uses the existing Dalos crypto stack — no new primitives needed.

**Setup at signup (or graduation):**
1. User has mnemonic M (24 BIP-39 words derived from initial signup, or chosen by user)
2. Dalos derivation: M → private key sk → CodexPrime account
3. Mnemosyne calls CodexPrime activation via Ouronet Gas Station — registers the account's public key pk on Stoa-chain
4. Server stores: `codex_prime_address` (the ouro address whose pk is on-chain)
5. CK is wrapped under password-derived key. Mnemonic is NOT stored on server (the user writes it down).

**Recovery flow (forgot password, has mnemonic):**
1. User clicks "Recover with mnemonic" in the Mnemosyne SPA
2. UI prompts for 24 words
3. Browser derives sk locally from the mnemonic
4. Server issues a recovery challenge (random nonce)
5. Browser signs the challenge with sk → produces Schnorr signature σ
6. Server reads CodexPrime's pk from Stoa-chain (using the stored `codex_prime_address`)
7. Server verifies σ against pk for the challenge — Schnorr verification on Dalos curve
8. ✅ → server grants password-reset capability
9. Browser derives CK from the mnemonic (separately from sk, e.g. via HKDF(M, "codex-key")) — OR — more cleanly: browser is granted access to download `encrypted_codex_key_facts` if it exists (grandma tier), OR refuses and forces wipe-and-restart for self-custody users who lost the password
10. User sets new password
11. Browser re-wraps CK under the new password-derived key → uploads new `encrypted_codex_key_password`
12. Codex content unchanged

Wait — there's a subtlety. If the user is SELF-CUSTODY (no facts wrapper), and they forgot the password, they need a way to RE-DERIVE CK from the mnemonic. Two designs:

**Design A — CK derived deterministically from mnemonic at signup:**
- CK = HKDF(mnemonic, "mnemosyne-codex-key-v1")
- Server stores `encrypted_codex_key_password` for normal login convenience
- If user loses password but has mnemonic: re-derive CK locally → decrypt codex → set new password
- Drawback: CK is deterministic from mnemonic, so anyone who guesses the mnemonic guesses CK. But guessing a 24-word mnemonic is already ~256 bits of work, so no real degradation.

**Design B — CK is random at signup, stored only under password wrapper:**
- CK = random 256-bit
- `encrypted_codex_key_password` = wrap(CK, KDF(password))
- If user loses password: codex content is gone forever; recovery just verifies identity for "you can create a new codex" but old contents are wiped
- Same model as Mnemonic-only without graceful preservation

**Design A is what we want for content preservation.** Locked.

### 4.3 Why the brain-wallet trap is sidestepped (important context)

Earlier in design discussion we considered deriving CK directly from personal facts ("email + name + DOB + secret word"). This is functionally identical to Bitcoin's failed 2013-2015 **brain wallet** scheme, where attackers drained thousands of BTC by brute-forcing low-entropy passphrase-derived keys.

**The fix in our design**: CK is HIGH-ENTROPY (256-bit random or 24-word-mnemonic-derived = 256 bits). The personal facts are only used as a WRAPPING KEY (to encrypt CK), not as the direct key derivation. An attacker who steals the DB and brute-forces the facts gets ONE user's codex, not all users. And the security still relies on the user's "secret word" being unguessable.

So:
- Facts-recovery is SAFER than brain-wallet because the facts only wrap a random key
- Facts-recovery is still WEAKER than mnemonic-recovery because the facts have lower entropy than 256-bit randomness
- Trade-off is appropriate for grandma-tier (convenience > absolute security); inappropriate for self-custody-tier (absolute security > convenience)

### 4.4 Grandma-tier (facts-recovery) flow

**Setup at signup:**
1. User provides 5-7 personal facts during signup (email, family name, given name, DOB, mother's language, secret word, etc. — exact schema TBD)
2. Facts canonicalized to a stable string (lowercase, trim, sort by field name, etc.)
3. `facts_hash = Argon2id(canonical(facts), salt)` → stored server-side for verification
4. `encrypted_codex_key_facts = encrypt(CK, KDF(canonical(facts)))` → stored server-side
5. Codex still encrypted under random CK (per §3.3)

**Recovery flow:**
1. User clicks "Recover with personal facts"
2. UI prompts for the facts (same schema)
3. Client canonicalizes locally → hashes → sends hash to server (NOT plaintext)
4. Server compares to stored `facts_hash` → match → returns `encrypted_codex_key_facts`
5. Client uses canonical(facts) to derive the wrapping key → unwraps CK
6. Decrypts codex
7. User sets new password
8. Client re-wraps CK under new password → uploads new `encrypted_codex_key_password`
9. Codex content preserved

**Security caveats:**
- If the personal facts schema is too predictable and the secret word is weak, an attacker with database access could brute-force CK. Mitigated by Argon2id at expensive parameters.
- User should be warned at signup: "Don't use your pet's name as the secret word. Don't use your birth year. Don't use anything publicly searchable about you."
- Documented value cap recommendation: "this tier is suitable for codices holding less than $X total value. For higher-value codices, please graduate to mnemonic recovery."

### 4.5 Guardian designation (extending recovery with secondary accounts)

Beyond CodexPrime, a user can designate **Guardians** — additional ouro accounts within their own codex (or external) that can ALSO prove ownership for recovery.

**Constraints (enforced by codex package v0.3.0):**
- Guardian MUST be a Standard Ouronet Account (`isSmart: false`)
- Guardian MUST be activated on Stoa-chain (otherwise no on-chain pubkey to verify against)
- Multiple Guardians allowed per codex (M-of-N not required at v0.1; any single Guardian's signature is sufficient for recovery)
- A Guardian's existence is publicly observable on-chain (the address is in `guardian_addresses` on the Mnemosyne server, but the on-chain account itself is normal)

**Recovery via Guardian:**
- Same flow as §4.2 self-custody recovery, but the user signs the challenge with the Guardian's private key instead of CodexPrime's
- Server verifies against the Guardian's on-chain pubkey
- Useful for users who lost CodexPrime mnemonic but remember a Guardian's seed (e.g., a custom-Dalos-seed account they created themselves)

**Designation flow (UI):**
- User must be authenticated (has codex unlocked)
- Selects an existing ouro account from their codex
- If isSmart === true → refused (Standard accounts only)
- If account not yet activated → warning shown, designation queued
- If activated → designation effective immediately
- Guardian's address added to `guardian_addresses` on server

### 4.6 What the admin (Ancient Holdings) can and cannot do

| Action | Self-custody user | Grandma-tier user |
|---|---|---|
| Decrypt the codex with server-side data alone | ❌ Impossible (no CK access) | ❌ Impossible without brute-forcing facts |
| Initiate a password reset on user's behalf | ❌ Impossible (would need to forge a Dalos Schnorr signature against on-chain pubkey — equivalent to breaking the curve) | ⚠️ Possible IF admin can trigger email reset AND user has weak personal facts |
| Tamper with the codex contents undetectably | ⚠️ Possible (could replace ciphertext with garbage) — but user would notice on next decrypt; mitigated by on-chain blob hash anchoring (v0.2+) | Same |
| Refuse to serve a user (denial of service) | ✓ Possible — but user has their mnemonic, can self-host a Mnemosyne instance, or migrate to another operator | Same caveat |
| Brute-force passwords from `password_verifier` | ⚠️ Theoretically possible but Argon2id makes weak passwords slow; strong passwords infeasible | Same |
| See WHICH codices exist | ✓ Yes (metadata isn't encrypted, only the codex payload) | Same |

The cryptographic guarantees are real. The operational risks (denial of service, observable metadata, weak password brute-force) are standard for the Bitwarden-pattern and accepted as part of the trade-off.

## 5. Onboarding, identity, & activation

### 5.1 The four identity modes

A user's Mnemosyne account can be authenticated via one or more identity modes. Modes are **cumulative** (a single account can have multiple modes active simultaneously, each independently sufficient for sign-in) and **switchable** (modes can be added or removed over the account's lifetime, with appropriate friction for removal). Two modes are unauthenticated-anonymous-friendly; two collect PII.

| # | Mode | Identifier | Auth | Recovery | Anonymity | Best for |
|---|---|---|---|---|---|---|
| 1 | **Privacy** | Username (any non-colliding string) | Password (Argon2id) | Mnemonic only | Maximum (zero PII) | Crypto-natives, privacy-focused |
| 2 | **Wallet** | CodexPrime ouro address (Ѻ./Σ.) | Sign challenge with codex private key (Schnorr-vs-on-chain-pubkey, the same trick that powers the recovery flow in §4.2) | Mnemonic (same key) | Maximum (zero PII) | **Existing OuronetUI users migrating in** — they already have a codex |
| 3 | **Convenience** | Email | Email + password | Mnemonic OR email-link reset | Medium (lightweight PII) | Power users wanting easy multi-device access |
| 4 | **Streamlined** | Phone number | Phone + password (NOT phone-only — see SMS / SIM-swap caveat below) | Mnemonic OR SMS code | Weakest (strong PII — phone is carrier-KYC'd) | Movie platform, grandma onboarding |

**Phone is the "must" for grandma-onboarding; email is optional add-on.** Per owner's locked decision: when grandma signs up via the movie platform, her primary identifier is phone (universal — everyone has one — and aligns with WhatsApp/Telegram/xPortal patterns). She can later add email as a secondary mode.

**Cumulative + switching semantics:**

- ADD a mode: requires current authentication + verification of the new mode (e.g., SMS code for new phone).
- REMOVE a mode: requires current authentication + 24-hour cooling-off period (prevents account takeover via single compromised mode). Removing the last identity mode (leaving only mnemonic recovery) requires an explicit "I understand I will only be able to log in via my mnemonic from now on" confirmation.
- The mode used to log in is decoupled from the codex's contents — the codex doesn't know what mode the user authenticated with.

**Graduation path** (the long-arc design intent):
- Grandma starts in Mode 4 (phone+password, movie platform)
- Over time, becomes more comfortable with crypto: adds Mode 2 (wallet-sign-in using her CodexPrime)
- Eventually decides she wants more privacy: removes Mode 4 (phone), keeps Mode 2 + password
- Net: she gradually self-custodies without ever losing access during the transition

### 5.2 Why SMS-only auth is rejected (security note)

SMS as a sole auth factor is well-documented as catastrophically vulnerable to **SIM swap attacks** in the crypto industry — attackers social-engineer carrier transfers of victim phone numbers, intercept SMS codes, drain accounts. Coinbase, Binance, and many crypto custodians faced waves of SIM-swap losses circa 2018-2021. **Mode 4 always pairs phone with password.** Phone is the identifier (and SMS may be one verification factor), but password is the security anchor. TOTP 2FA (authenticator apps) is the future stronger-2FA path; SMS-2FA is explicitly NOT recommended.

### 5.3 Herotag — on-chain registration

The herotag (a globally-unique human-readable handle, e.g. `@mihai`) is a public identifier that resolves to a specific ouro address within the user's codex. **The herotag registry lives on Stoa-chain**, not in Mnemosyne's database. This is a meaningful architectural choice (locked 2026-05-27 refinements pass) — see §5.3.4 below for why.

#### 5.3.1 The on-chain registry

A small immutable Pact module on Stoa, conceptually:

```pact
(module mnemosyne-herotag IMMUTABLE-GOV
  ;; Immutable governance keyset — module logic frozen forever
  (defcap IMMUTABLE-GOV () (enforce false "no-governance-by-design"))

  (defschema herotag-row
    name: string                ; lowercased, no @ prefix
    target-address: string      ; the Ѻ. address the herotag resolves to
    claimed-by-pubkey: string   ; pubkey that signed the claim
    claimed-at: time)

  (deftable herotags: {herotag-row})

  (defcap CLAIM (target-address: string)
    "Caller must control the target address's account guard."
    (let ((acct-guard (at 'guard (coin.details target-address))))
      (enforce-guard acct-guard)))

  (defun claim-herotag (name: string target-address: string)
    (enforce (validate-format name) "invalid herotag format")
    (with-capability (CLAIM target-address)
      ;; (insert) FAILS if name already exists — uniqueness is database-level,
      ;; atomic, race-free at the consensus layer
      (insert herotags name { ... })))

  (defun resolve-herotag (name: string)
    ;; Public read — anyone can resolve any herotag
    (read herotags name)))
```

#### 5.3.2 Claim flow

1. User in the Mnemosyne SPA (or any consumer with herotag-claim UX) clicks "Claim herotag"
2. UI shows their accounts: CodexPrime highlighted as default; user can select any other **Standard, activated** account in the codex as the target
3. User types desired name; UI calls `(mnemosyne-herotag.is-available "mihai")` via a free chain read
4. If available, user signs a Pact tx invoking `claim-herotag` — signed with the target account's keys to satisfy `(CLAIM target-address)` capability. Gas sponsored by Ouronet Gas Station (Mnemosyne or partner app pays)
5. Pact's `(insert)` is atomic: if two users race for the same name, exactly one succeeds at consensus
6. Codex's `herotag` field (see §9, codex package v0.3.0) is populated client-side once the chain confirms

#### 5.3.3 Constraints (enforced both on-chain and in the codex package)

- `targetOuroAddress` must be a **Standard Ouronet Account** (`isSmart: false`)
- Account must be **activated on-chain** at claim time (otherwise `enforce-guard` fails since there's no chain record to enforce against)
- Name must satisfy format: lowercased, alphanumeric + dots + hyphens, 3-30 chars
- Name must be globally unique (enforced cryptographically by Pact's `insert`)

#### 5.3.4 Why on-chain (not Mnemosyne DB)

| Property | Mnemosyne DB (rejected) | Stoa-chain Pact module (chosen) |
|---|---|---|
| Resolution survives Mnemosyne dying | ❌ | ✓ — any Stoa node serves the read |
| Uniqueness enforcement | App-level (race conditions possible) | Database-level via `(insert)` (atomic, race-free) |
| Authoritative source | Mnemosyne owns the truth | Chain owns the truth — Mnemosyne caches |
| Resolution cost | Mnemosyne API call | Free chain read |
| Auditability | Requires DB access | Public on-chain history |
| Composability — other Stoa apps can use herotags | ❌ Mnemosyne-only | ✓ Open to any chain reader |
| Operator (Mnemosyne) abuse risk | Could in principle re-target a herotag | None — `with-capability (CLAIM ...)` makes this cryptographically impossible |

The on-chain approach is a strict improvement. The codex package's `herotag` field carries the same shape regardless, but its `onChainTxId` / `onChainConfirmed` fields (see §9.1.6) reflect the chain source of truth.

#### 5.3.5 Immutability — why the target can't be changed later

The whole value of a herotag is **persistent identity for cross-user references**. If `@mihai` re-targets between accounts over time:
- Address book entries become trust-broken
- Other users' historical "send to @mihai" history becomes ambiguous
- Movie platform's herotag-display in user lists becomes inconsistent

**Rule (enforced by the Pact module — no `update` function exists, only `insert`):** once claimed, a herotag's target is **immutable**. If the user needs a different account as their public identity, they claim a new herotag (different name). They cannot reuse `@mihai` for a different target. xPortal (MultiversX) follows the same rule for the same reasons.

**Edge case — compromised target account:** if the targeted account's keys leak, the herotag is "stuck" pointing to a compromised account. Mitigation: user runs `rotateGuard` on the compromised account (existing OuronetUI flow). The *address* stays the same; the *signing keys* change. Herotag stays valid AND the account is no longer compromised.

### 5.3 Why activation matters for v0.1

### 5.7 Codex Identity — on-chain registration via `mnemosyne.CODEX` (2026-05-28 Mnemosyne-Ledger pass)

Every Mnemosyne-managed codex has a **Codex Identity**: an Apollo public key derived from user-supplied seed input (1-256 words from the Dalos 256-glyph character set, or direct bitstring/base49 entry, or system-generated 12-word default). The Codex Identity is the codex's user-facing identifier (`Π.<short-form>` address format) and is **immutable for the lifetime of the codex** — choosing it is a deliberate user-visible step at signup.

Alongside the Apollo identity, every Mnemosyne-bound codex auto-generates a `PrimePureKey`: a Kadena Ed25519 keypair stored *inside* the encrypted codex blob. The PrimePureKey serves as the Pact-native authorization guard for all on-chain codex management operations (Apollo signatures cannot satisfy Kadena keyset guards — different curve, no native Pact verifier). See `stoa-js/packages/ouronet-codex/docs/v0.3.0-design.md` §2.4.bis for the full PrimePureKey interface.

#### 5.7.1 The on-chain registry

A new Pact module on Stoa:

```pact
(namespace 'mnemosyne)

(module CODEX IMMUTABLE-GOV
  ;; (defcap IMMUTABLE-GOV ...) — frozen governance, no admin upgrade path

  (defschema codex-identity-row
    codex-public-id:    string         ;; immutable: Apollo pubkey (hex, 64 chars)
    codex-guard:        guard          ;; immutable: Kadena keyset from PrimePureKey
    registered-at:      time           ;; immutable
    registered-by:      string         ;; immutable (client identifier)
    arweave-tx-id:      string         ;; mutable: latest L2 backup pointer (or "")
    arweave-updated-at: time           ;; mutable
    arweave-backup-count: integer)     ;; mutable (counter for SPA display)

  (deftable codex-identities:{codex-identity-row})  ;; key = apollo-address

  (defschema codex-arweave-history-row
    codex-public-id:string
    arweave-tx-id:  string
    uploaded-at:    time)

  (deftable codex-arweave-history:{codex-arweave-history-row})
  ;; key = "<apollo-address>|<arweave-tx-id>" (composite for v0.1 simplicity)

  (defun register-codex-identity (apollo-address codex-public-id codex-guard registered-by)
    ;; one-shot: insert throws on existing apollo-address
    ;; capability-enforced: apollo-address must equal codex-public-id
    ...)

  (defun update-arweave-pointer (apollo-address arweave-tx-id)
    ;; authorized by CODEX-OWNER capability (enforce-guard on codex-guard)
    ;; atomic: inserts history row + updates main-table mutable fields
    ...))
```

Full module spec in `Mnemosyne/docs/handoffs/01-mnemosyne-codex-pact-module.md` (handoff document for the Pact implementer).

#### 5.7.2 Registration flow

1. User completes signup → Mnemosyne SPA has their chosen Apollo seed input + a fresh codex
2. SPA derives Apollo keypair from seed input (HKDF; see open-decisions doc)
3. SPA generates fresh PrimePureKey via `kickstartCodex({ generatePrimePureKey: true })` (codex package)
4. SPA derives Kadena keyset from the PrimePureKey's public key → `codex-guard`
5. SPA calls `codex.buildRegisterCodexIdentityTx({ apolloAddress, codexPublicId, codexGuard, registeredBy: "mnemosyne-spa@1.0.0" })`
6. Returned unsigned Pact tx is signed with the user's just-generated PrimePureKey (private half is in memory; gets encrypted under CK before the codex is written to L1)
7. Tx submitted to Stoa node via Ouronet Gas Station sponsorship (negligible user cost)
8. On confirmation, codex is written to L1 (Mnemosyne DB) with the PrimePureKey embedded
9. User is on Mnemosyne; Codex Identity is registered on-chain; Arweave backup is opt-in from here on

#### 5.7.3 Why a separate registration step (not just "codex on Mnemosyne implies registered")

The on-chain registration is what makes the trust anchor work. Without it, an attacker holding the same Apollo seed input (via collision or social engineering) could overwrite a user's L2 Arweave pointer to point at their own decoy blob. The chain entry's `codex-guard` is what cryptographically binds "this Apollo identity" to "this PrimePureKey" to "this user's right to update the L2 pointer."

Apollo collision risk is genuinely tiny (the seed input space is massive), but the registration is cheap and irreversible — there's no reason not to do it.

#### 5.7.4 What chain-side validation does (and doesn't) protect

| Threat | Protected? | How |
|---|---|---|
| Two users claiming the same Apollo address | ✅ | Pact `insert` enforces uniqueness on `apollo-address` key |
| Pointer overwrite by non-owner | ✅ | `CODEX-OWNER` capability enforces `codex-guard` |
| Invalid `arweave-tx-id` format | ✅ | `validate-arweave-tx-id` checks length + base64url charset |
| Pointing `arweave-tx-id` at a junk Arweave tx | ❌ | Only verifiable client-side (user fetches + tries to decrypt) |
| Replay of an old valid `arweave-tx-id` | ❌ | Pact `insert` on history table prevents same-tx-id-twice, but doesn't prevent reverting to an earlier backup. The "latest" pointer is whatever was most recently written. |
| Module governance compromise (e.g. malicious upgrade) | ✅ | `IMMUTABLE-GOV` capability `enforce false`s — no upgrade path exists |

The "junk Arweave tx" case is acceptable because the user is the only party who would push to their own codex pointer (CODEX-OWNER gates this). If they shoot themselves in the foot, they recover by uploading a fresh good backup. No protection of the user from themselves needed at the chain layer.

## 7. Multi-device & multi-consumer behavior

### 7.1 The unified-codex promise

The whole point of Mnemosyne: **one codex per user, accessible from every consumer**. When a user adds an account in OuronetUI on their laptop, that account appears in OuronetUI on their phone, in the AH admin UI (if they're an admin), in the movie platform's wallet view (if it has one), etc. Single source of truth.

### 7.2 Conflict resolution

When two consumers (or two instances) write to the same codex simultaneously, we need a conflict policy. Three options:

| Policy | Pros | Cons | Verdict for v0.1 |
|---|---|---|---|
| **Last-writer-wins (LWW)** | Simplest; matches user mental model 80% of the time | Can silently overwrite the other write | ⚠️ Acceptable if uncommon |
| **Optimistic concurrency (version counter)** | Detects conflicts; client retries on stale write | Requires UX for "conflict — your changes weren't saved, please retry" | ✓ Recommended for v0.1 |
| **Vector clocks / CRDTs** | Auto-merges without user-visible conflicts | Complex implementation; harder to reason about | Too heavy for v0.1; consider for v0.3+ |

**Recommended v0.1 policy:** optimistic concurrency. Server tracks a monotonically increasing `version` per codex. Client includes `if-version: N` header on PUT. If server's current version is > N, returns 409 Conflict. Client must re-fetch, re-apply local changes, retry.

This means clients should batch their writes (debounce small changes; PUT a complete codex blob at a sensible interval, not per-key-stroke).

### 7.3 Per-consumer settings namespacing

Today's UiSettings has OuronetUI-specific fields mixed with generic codex settings. With multiple consumers writing to the same codex, this becomes a collision risk.

**v0.3.0 codex package change (see §9):** introduce `consumerSettings: Record<string, IConsumerSettings>` registry, namespaced by consumer name. OuronetUI writes to `consumerSettings["OuronetUI"]`. AH hub to `consumerSettings["AncientHoldings"]`. Movie platform to `consumerSettings["MoviePlatform"]`. No collisions.

### 7.4 What lives in the codex vs what stays app-local

A clean rule, drawn here while it's clean:

| Category | Lives in codex | Lives in app |
|---|---|---|
| Identity-adjacent (keys, addresses, settings) | ✓ | |
| Cross-consumer-useful (address book, watch list) | ✓ | |
| Auth/signing preferences | ✓ | |
| Personalization (theme, language, units) | ✓ | |
| Per-consumer UI preferences (e.g. selectedNode for OuronetUI) | ✓ (namespaced under consumerSettings) | |
| Application data (movie history, pool position cache) | | ✓ |
| Large or churny state (transaction history cache, balances) | | ✓ |
| Anything that doesn't make sense in another consumer | | ✓ |

Codex stays meaningful and reasonably small. App-local state stays app-local.

## 8. Decentralization layers (most deferred, but identity + opt-in backup ship in v0.1)

The v0.1 core is centralized for the *hot path* of codex blob storage (Mnemosyne PostgreSQL is the source of truth for the live encrypted ciphertext). But three decentralization pieces ship in v0.1:

1. **Herotag registry is on-chain** in an immutable Pact module on Stoa (see §5.3). The chain-based herotag is a strict improvement over a Mnemosyne-DB-based one.
2. **Codex Identity registry is on-chain** in `mnemosyne.CODEX` (see §3.4 + §5.5 below). Every codex's Apollo public key and Kadena guard get registered on Stoa at signup.
3. **Per-codex Arweave backup** is an opt-in v0.1 feature paid by the user (see §8.2 below).

Further decentralization layers are progressive enhancements deferred to later releases.

### 8.1 ~~v0.2 — Arweave snapshot layer (L2 pattern)~~ REJECTED (2026-05-28 Mnemosyne-Ledger pass)

The original v0.2 plan called for Mnemosyne to batch-snapshot the entire encrypted vault state and write it to Arweave on a schedule (hourly cron) for all users. This was **rejected during the Mnemosyne-Ledger refinements pass** in favor of per-codex opt-in (§8.2 below).

Reasons for rejection (see §3.4.3 for the full discussion):
- HNDL exposure of users who never opted in to public-ledger storage
- Operational complexity (cron, retry, billing, size-limit handling)
- Cost to Mnemosyne ($720/mo at 1000 users for a service most users wouldn't request)
- GDPR-impossible "delete my Arweave copy" promise
- Trust model leak (users couldn't verify per-upload that the right blob went up)

The opt-in model (§8.2) preserves all the decentralization benefits while making each user's privacy-vs-durability trade-off explicit.

### 8.2 v0.1 — Per-codex Arweave pinning (user-paid via Bundlr/Irys) — SHIPS IN v0.1

- **Available to all tiers as opt-in** (Adyton/Naos/Pronaos/Agora users can all trigger). The tier model stays orthogonal to the storage destination — tiers are about recovery factors, not about backup location.
- **User-pays via Bundlr/Irys.** SPA integrates the Bundlr/Irys SDK; user funds upload with card or crypto on-ramp. Mnemosyne stays out of the billing path (no Stripe account, no subscription system, no refund handling). Cleanest payment model.
- **Workflow:**
  1. User clicks "Backup to Arweave" in the Mnemosyne SPA.
  2. SPA calls `codex.exportEncryptedSnapshotForArweave()` → gets ciphertext + size + sha256.
  3. SPA displays cost (in AR + USD equivalent fetched from Bundlr).
  4. User confirms payment → SPA uploads via Bundlr session → Arweave returns `arweave-tx-id`.
  5. SPA calls `codex.buildUpdateArweavePointerTx(...)` → unsigned Pact tx.
  6. Tx is signed with the user's unlocked PrimePureKey (already in the in-memory codex state via the package's CodexSigningStrategy).
  7. Signed tx submitted to a Stoa node (gas-station-sponsored, ~minimal user cost).
  8. On confirmation, SPA calls `codex.recordArweaveBackupLocally(...)` → entry added to local history cache for SPA display.
- **Version history.** Stoa stores ALL Arweave upload tx-ids in the `codex-arweave-history` table, keyed by `<apollo-address>|<arweave-tx-id>`. Main table `codex-identities.arweave-tx-id` holds only the latest. SPA can enumerate full history (sorted by `uploaded-at`) by reading the history table; user can restore from any prior backup, not just the latest.
- **Codex package boundary.** ALL Arweave + chain-pointer logic lives in `@stoachain/ouronet-codex` (see §9.1.6). The Bundlr/Irys SDK is NOT a package dependency — the SPA supplies the uploader callback. The package stays chain-client-agnostic for the chain-tx submission too (SPA owns the Pact node integration).

### 8.3 v1.0 — Optional on-chain codex blob storage (deferred)

The v0.1 design intentionally keeps the encrypted blob OFF chain. Reasons:
- Stoa tx size limit (~75KB raw) excludes power-user codices entirely.
- Gas costs scale with blob size on every codex mutation; prohibitive for casual users.
- Public-ledger permanence creates HNDL exposure that the opt-in Arweave path already provides (with the benefit of pay-once-store-forever rather than per-mutation gas).

A future v1.0 *might* explore per-field on-chain storage (Pact tables per entity kind: seeds, accounts, address book, etc.) for users who want the strongest possible decentralization story. This is genuinely separate work and not blocked by the v0.1 design — the `mnemosyne.CODEX` module's identity layer would extend with content tables; the L1 (Mnemosyne DB) layer would become a cache rather than a source of truth.

Timeline: revisit only after v0.1 has real users + we understand actual codex-content access patterns.

### 8.4 SPA permaweb deploy (independent of the above)

- Mnemosyne SPA deployed to Arweave per major release (~$10-30 per deploy).
- Users can access the SPA at `arweave.net/<tx-id>` as a censorship-resistant fallback.
- Same pattern applies to OuronetUI's SPA — monthly permaweb deploys make OuronetUI itself censorship-resistant at the frontend level.

### 8.5 What "decentralized" actually means after these layers

- **v0.1 (without Arweave opt-in)**: centralized SaaS for blob, on-chain only for identity + herotag. Single point of failure for codex data.
- **v0.1 (with Arweave opt-in)**: opt-in users have full censorship-resistant recovery — Mnemosyne can die, the user still recovers via L3 → L2 → CK decrypt.
- **v1.0 (if shipped)**: optional on-chain content storage for users who want it. Strongest possible decentralization; pays gas + permanence costs.
- **At v1.0**: Mnemosyne is *operationally* decentralized for opted-in users — anyone running a Stoa-chain client can read/verify identity + recover backups. AH-as-an-operator becomes "the largest operator," not the only one.

This is the Bitcoin/Coinbase pattern applied to codex management. Bitcoin (decentralized network) plus Coinbase (centralized custodian for non-crypto users). Both layers coexist; neither depends on the other for survival.

### 10.1 Technical

1. **Backend framework.** Next.js (matches AH hub) / Express / Hono / Fastify — pick one based on team familiarity + ecosystem.
2. **Database engine.** SQLite (single-server, matches AH hub) vs Postgres (scales horizontally if needed later). SQLite is simpler at v0.1; Postgres avoids a migration if we grow.
3. **Auth library.** Roll our own with Argon2id (~1 week, fewer deps) vs Lucia (lightweight, modern) vs Auth.js (more features, more bloat) vs Supabase (fastest to ship, vendor lock-in). Need to evaluate.
4. **DB encryption at rest.** Server-side disk encryption (SQLCipher / Postgres TDE) gives defense-in-depth but doesn't help against a compromised process. Discuss.
5. **Session token strategy.** JWT (stateless, no DB lookup per request) vs DB-backed sessions (revokable, can be invalidated server-side). JWT is simpler; revocation is harder.
6. **Multi-device sync conflict resolution.** Optimistic concurrency (recommended in §7.2) is the v0.1 answer. UX for the rare conflict case TBD.
7. **API rate limiting strategy.** Per-user / per-IP / both. Prevents brute-force on the recovery facts endpoint specifically.
8. **2FA option.** Optional TOTP-based 2FA for additional account security. Probably v0.2+; not required for v0.1.
9. **SMS provider** (only relevant if Streamlined mode ships in v0.1). Twilio / MessageBird / AWS SNS / Vonage. Cost ~$0.01-0.05/SMS. Affects ops budget.
10. **Herotag Pact module deployment.** Who deploys (which account), at what time relative to Mnemosyne launch, on testnet vs mainnet, how do we manage the immutable-governance keyset securely so it's truly unusable (e.g., publish the un-satisfiable predicate publicly so anyone can audit). See §5.3.1.
11. **Arweave snapshot frequency + batching strategy** (deferred to v0.2 planning — user's instinct: weekly server-side batch, server pays, possibly skip-if-unchanged algorithm. See top of doc / 2026-05-27 conversation for context).

### 10.2 Product

1. **Pricing model.** Free for grandmas (via partner-app gas station)? Paid Arweave-permanence tier? Bundled with Mnemosyne API access for partner apps? Affects business model + may inform regulatory posture.
2. **Movie-platform integration spec details.** Once we know the movie platform's tech stack, design the partner-app onboarding API in detail. For now, sketched in §6.3.

## 12. Risks & deferred topics

### 12.1 Regulatory posture

The owner's stated vision is **jurisdictionless** ("system that works and helps consumers achieve their goal... no entity should dictate how it can be used"). The technical design respects this — the cryptographic guarantees make Mnemosyne **technically non-custodial** at the self-custody tier. But the *operator* (Ancient Holdings) still has a legal posture to think about for the SaaS layer, especially for grandma-tier users where the operator's role is more involved.

**Plan:** discussed in design conversation but DEFERRED. Approach when scoping v0.1 launch:
- For self-custody users: the operator demonstrably cannot access user keys. This is the strongest defensible legal position. Document it clearly in ToS.
- For grandma-tier users: the operator has theoretical recovery assistance capability. This is custodial-flavored. Probably needs MiCA CASP consideration if operated from the EU.
- Long-term: convert Ancient Holdings to a foundation (already planned per owner's note) to take the entity-as-operator dimension off the table.

Owner explicitly does NOT want this to delay technical work. Counsel engagement is parallel to engineering.

### 12.2 Brain-wallet trap

We considered (and rejected) deriving the codex key directly from personal facts. Bitcoin's history with brain wallets is conclusive — direct human-input-derived keys are catastrophically vulnerable. Our design uses facts only as a WRAPPING factor around a random key. This is meaningfully safer but still weaker than mnemonic-based recovery. Clearly documented in §4.3-4.4.

### 12.3 Multi-implementation governance

Not relevant at v0.1 (single implementation, Ancient Holdings operates). Becomes relevant if at v1.0+ we open up the protocol for other operators. Punt until then.

### 12.4 Operator key compromise

If Mnemosyne's server keys (TLS, JWT signing, partner-app validation) are compromised, the attacker can issue valid sessions and trigger Arweave snapshots with fake data. Mitigations:
- Sensitive operations (Arweave snapshot signing, partner-app validation) use HSM-stored keys
- Audit log to detect anomalous activity
- Out-of-band notification to users for high-risk operations (password resets, Guardian designations, etc.)

These are operational concerns, not architectural. Standard SaaS security hygiene.

### 12.5 Loss of mnemonic AND password

For self-custody users: codex is gone. Documented at signup with strong UX.
For grandma-tier users: facts-recovery is the safety net. If facts are also lost: codex is gone.

This is the inherent trade-off of the design. Cannot be solved without making the operator custodial (which the owner explicitly opposes).

### 12.6 Related projects — SureMark Digital (sidebar)

[SureMark Digital](https://www.suremark.digital) is run by **Stuart Haber** and **W. Scott Stornetta** — the literal inventors of blockchain (their 1991 paper "How to Time-Stamp a Digital Document" is the most-cited paper in Satoshi's Bitcoin whitepaper). Owner has direct relationship with Haber; was a consultant on the original Chainweb creation.

**What SureMark does:** cryptographic identity verification for combating AI-deepfake / impersonation attacks. Two products: SureCircle (one-click identity verification before sensitive actions — e.g., "is this really the CFO requesting the wire?") and Verified Web (cryptographic proof of authorship for public figures' statements).

**Architectural parallel with Mnemosyne:**

| Layer | SureMark | Mnemosyne |
|---|---|---|
| Purpose | Prove "this action/message is authentically from person X" (provenance) | Store and serve person X's crypto codex across apps (state) |
| Architecture | Centralized registry + client-side keypair + verification protocol | Centralized SaaS + client-side encryption + multi-app access |
| Primitive | Cryptographic signature against registered pubkey | Cryptographic signature against on-chain pubkey (Schnorr-Dalos in our case) |
| Founders | Haber + Stornetta (blockchain inventors) | (us) |

**Significance:** the fact that Haber + Stornetta independently arrived at the same architectural pattern (cryptographic identity registry with client-side keys) is meaningful validation of the broad design space. They could have built anything; they built this shape. We're not inventing — we're picking up a well-validated pattern.

**Future integration possibilities (not v0.1, but worth holding in head):**
- A herotag could optionally be **verified via SureMark** — "@mihai ✓ verified by SureMark, you can trust signed messages from this account"
- SureMark could potentially use Mnemosyne as the codex layer for storing user identity keypairs
- Cross-pollination through Haber's existing relationship with the owner

For v0.1: noted, not pursued. Revisit when we have something to show Stuart.

## 13. Glossary

| Term | Meaning |
|---|---|
| **Codex** | An encrypted multi-wallet store containing kadena seeds, ouro accounts, pure keypairs, address book, watch list, settings. Source-of-truth for a user's ouronet identity. Defined by `@stoachain/ouronet-codex`. |
| **CodexPrime** | The first ouro account in a codex, derived from the same mnemonic as the Prime Codex Seed. Structurally unremovable. The canonical on-chain identity of the codex. |
| **Prime Codex Seed** | The kadena seed that birthed the codex (the original 12-or-24-word mnemonic). Structurally unremovable. |
| **Guardian** | A non-CodexPrime ouro account designated as an additional recovery factor. New in codex package v0.3.0. |
| **Herotag** | A user-chosen globally-unique human-readable handle (e.g. `@mihai`) registered on-chain in a Pact module, resolving to a specific ouro account in the user's codex. Immutable target. New in codex package v0.3.0. |
| **CK** (Codex Key) | The random 256-bit AES key that encrypts a codex blob in Mnemosyne. Wrapped by password and/or facts. |
| **Mnemosyne** | This project. Centralized SaaS for codex storage with cryptographic non-custodial guarantees. |
| **SaaS** | Software as a Service. Cloud-hosted software accessed via browser/API. |
| **SPA** | Single Page Application. JavaScript app that loads once and re-renders dynamically. |
| **Self-custody tier** | Recovery via mnemonic only. Admin cannot help. Strong guarantee. |
| **Grandma tier** | Recovery via personal facts wrapping the random codex key. Admin has weaker (still cryptographically bounded) recovery capability. Easier UX. |
| **Identity Mode** | One of four ways to authenticate to Mnemosyne: Privacy (username), Wallet (sign with codex keys), Convenience (email), Streamlined (phone). Cumulative — an account can have multiple modes active. See §5.1. |
| **Privacy Mode** | Identity mode 1: username + password. Zero PII. Recovery: mnemonic only. |
| **Wallet Mode** | Identity mode 2: sign in by signing a challenge with CodexPrime keys (Schnorr-vs-on-chain-pubkey). Zero PII. The natural mode for migrating existing OuronetUI users. |
| **Convenience Mode** | Identity mode 3: email + password. Light PII. Recovery: mnemonic OR email link. |
| **Streamlined Mode** | Identity mode 4: phone + password. Heavier PII (carrier-KYC'd phone). Recovery: mnemonic OR SMS. The grandma / movie platform path. |
| **Schnorr signature** | The signature scheme on the Dalos elliptic curve. Used for the on-chain-pubkey recovery verification AND for Wallet-Mode sign-in. |
| **SIM swap** | Attack where attacker socially-engineers a carrier into transferring victim's phone number to attacker-controlled SIM, then intercepts SMS codes. Why SMS-only auth is rejected for crypto custody. |
| **Ouronet Gas Station** | The existing `STOA_AUTONOMIC_OURONETGASSTATION` account that sponsors gas for OuronetUI users. Extended scope for Mnemosyne: sponsors CodexPrime activations during signup AND herotag claims. |
| **Partner app** | A consumer service (movie platform, etc.) that onboards users through Mnemosyne. Has its own gas-station budget for sponsoring activations + herotag claims. |
| **Permaweb** | Arweave's "permanent web hosting" — static SPAs deployed once, stay online indefinitely via Arweave's endowment-funded storage model. |
| **L2 / Layer 2** | "Layer 2" — an off-chain or off-decentralized-storage system that periodically commits state to a more decentralized substrate. Mnemosyne's Arweave snapshot model is L2-on-Arweave. |
| **Bitwarden pattern** | Centralized SaaS + client-side encryption + multi-device sync. Server stores opaque ciphertext; cannot decrypt. The architectural template Mnemosyne follows. |
| **SureMark Digital** | Stuart Haber + Scott Stornetta's identity-verification company (anti-deepfake). Architecturally parallel to Mnemosyne but for provenance not state. See §12.6. |
