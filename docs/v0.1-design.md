## 1. Vision

### 1.1 The two problems being solved

**Problem 1 — Origin fragmentation.** The OuronetUI codex today lives in browser `localStorage`, tied to the page's origin. When OuronetUI migrates from `devwallet.ouronetwork.io` to `wallet.ouro.network`, each user's codex stays trapped under the old origin. Same when AncientHoldings's admin UI moves to `codex.ancientholdings.eu`. Users would have divergent codices per origin with no automatic way to unify them. The same problem appears when a user wants to use OuronetUI on both desktop and mobile — separate localStorage means separate codices.

**Problem 2 — Transparent codex provisioning for non-crypto users.** Future consumer services in the StoaOuronet ecosystem (most concretely: an upcoming movie-streaming platform) will let users register with `{email, password}`. The vision is that those users *transparently* get an ouronet address as a side effect — they don't need to know what a codex is, what a mnemonic is, or what self-custody means. They register for movies; the system happens to have minted them a wallet they can use elsewhere later (e.g. on OuronetUI, with the same credentials, their codex is already there). When ready to learn, they can "graduate" to self-custody.

### 1.2 What Mnemosyne is

A **centralized SaaS** that stores user codices encrypted-at-rest, serves them to consumer services (OuronetUI, AncientHoldings hub, the movie platform, anything else) under a unified credential, and enables transparent codex provisioning for non-crypto users. Architecture is patterned after **Bitwarden** — well-proven, well-understood, shippable in weeks rather than years.

### 1.3 What Mnemosyne is NOT

- **NOT a decentralized protocol.** Earlier exploration considered building a Bitcoin-like protocol with cryptographically-decentralized governance. That's a multi-year research project; out of scope. Decentralization happens in layers via optional Arweave snapshots and on-chain Stoa pointers (see §8) — but the v0.1 core is a centralized service.
- **NOT a custodial wallet service in the regulatory sense.** The cryptographic design makes Mnemosyne **technically non-custodial**: the operator cannot decrypt user codices for self-custody users. For grandma-tier users, the operator can theoretically assist with recovery via wrapped recovery factors — clearly documented opt-in.
- **NOT a replacement for the `@stoachain/ouronet-codex` package.** It's a new *adapter* for the existing package — `MnemosyneCloudAdapter` — alongside the existing `LocalStorageCodexAdapter` and `MemoryCodexAdapter`. The codex data model, signing pipeline, and all existing surfaces are unchanged.
- **NOT an identity provider for non-codex purposes** (yet). Initial scope: codex storage + retrieval. SSO-style "Sign in with Mnemosyne" is a future possibility, not a launch goal.
- **NOT a regulated entity** at v0.1. Operator (Ancient Holdings) intends to convert to a foundation later; the technical design is jurisdictionless-by-construction (no operator privileged access to user keys), but the *operator* still has to think about its legal posture for the SaaS layer. See §12.

### 2.2 Why split static frontend / dynamic backend onto different domains

- **Static frontend can be Arweave-permaweb-deployed.** Vite produces a `dist/` folder; that bundle posts to Arweave one-time and stays online forever. Even if Cloudflare drops the `mnemosyne.ancientholdings.eu` domain, users can hit `arweave.net/<tx-id>` and the SPA still works (it just talks to the same backend API).
- **Backend cannot be permaweb** — it has a database, session state, dynamic queries. Must be a regular server.
- This is a common pattern for "decentralization-grade" web apps that need some centralized state. OuronetUI itself should follow the same pattern eventually (its SPA can be permaweb-deployed; the Stoa nodes it talks to are already decentralized).

### 2.3 Tech choices (v0.1)

| Layer | Choice | Rationale |
|---|---|---|
| Frontend framework | React + Vite | Matches OuronetUI's stack; team knows it; SPA-friendly |
| Backend framework | TBD (Next.js / Express / Hono) | Open question — see §10 |
| Database | TBD (Postgres / SQLite) | Open question — see §10 |
| Auth | TBD (roll-our-own Argon2id / Lucia / Auth.js / Supabase) | Open question — see §10 |
| Password hash | Argon2id (parameters tuned for ~500ms) | Industry standard for password hashing |
| Codex blob encryption | AES-256-GCM | Authenticated encryption; standard |
| Key derivation (password→key) | Argon2id-KDF or HKDF over Argon2id-hash | Standard |
| Signing primitive (recovery) | Schnorr signatures on Dalos curve | Already exists in `@stoachain/stoa-core/signing` |
| Hosting | Hetzner VPS (same as AH hub) | Owner's existing infra |
| Domain | `mnemosyne.ancientholdings.eu` + `api.` subdomain | Settles the consumer integration story |
| Permaweb | Arweave (for SPA only) | Per-release deploy, ~$10-30 per major release |

## 3. Storage model

### 3.1 What the server stores per user

```
users table:
  id                           (UUID)
  email                        (unique, indexed)
  password_verifier            (Argon2id hash of password, with salt)
  codex_prime_address          (ouro address of CodexPrime, for on-chain pubkey lookup)
  guardian_addresses[]         (additional ouro accounts designated as Guardians)
  created_at, last_login_at

codex_blobs table:
  user_id                      (FK)
  encrypted_codex              (AES-256-GCM ciphertext of the codex JSON)
  encrypted_codex_key_password (codex encryption key wrapped by password-derived key)
  encrypted_codex_key_facts    (codex encryption key wrapped by facts-derived key) [grandma tier only]
  facts_hash                   (Argon2id hash of canonical(personal facts)) [grandma tier only]
  schema_version               (data schema version of the codex JSON inside)
  version                      (monotonic counter for optimistic concurrency)
  last_updated_at, last_updated_by_device

audit_log table:
  user_id, event_type, ip, user_agent, timestamp
  events: signup, login, password_reset, recovery_attempt, codex_update, ...
```

**Key invariant:** the server stores ONLY ciphertexts and verifiers. It cannot decrypt any user codex with anything it holds. Decryption requires the user-side password (or recovery facts, for grandma tier).

### 3.2 What flows over the wire

- On `GET /codex`: server returns `encrypted_codex` + `encrypted_codex_key_password`. Client decrypts locally.
- On `PUT /codex`: client uploads new `encrypted_codex` (re-encrypted under the same key). Server doesn't see plaintext.
- Server never sees the codex contents, the password, or the codex key in plaintext.

### 3.3 Encryption envelope (the "wrapping" model)

Codex contents are NOT directly encrypted with the password. Instead:

1. A **random Codex Key (CK)** is generated at signup (256-bit random).
2. The codex JSON is AES-256-GCM encrypted with CK → stored as `encrypted_codex`.
3. CK is then **wrapped** (encrypted) with a key derived from the user's password → stored as `encrypted_codex_key_password`.
4. For grandma tier, CK is *also* wrapped with a key derived from canonicalized personal facts → stored as `encrypted_codex_key_facts`.

Why this matters:
- **Password change preserves codex content.** When the user changes password, only `encrypted_codex_key_password` is re-computed. The much larger `encrypted_codex` blob is untouched, because CK didn't change. Same applies to all the contents inside (Address Book, watch list, settings, every seed/account) — preserved verbatim.
- **Recovery preserves codex content.** Same reason. Recovery via mnemonic or facts unwraps CK without changing it.
- **Multiple unwrap paths possible without re-encrypting the blob.** Useful for adding Guardians, multi-device keys, etc.

> Note: I had been using "CEK" (Codex Encryption Key) in earlier discussion. Renaming to **CK** (Codex Key) here for cleaner terminology — there's no real semantic difference but "CEK" was a Claude-coined term that doesn't need to stick.

## 4. Cryptographic primitives & custody model

### 4.1 Two tiers, user picks at signup

| Tier | Recovery method | Cryptographic guarantee | Best for |
|---|---|---|---|
| **Self-custody** (default for crypto-natives) | Mnemonic only (proves ownership via Schnorr-sig vs on-chain CodexPrime pubkey) | Strong — admin cryptographically *cannot* help. Recovery requires same entropy as the original seed. | OuronetUI users, valuable codices, paranoid users |
| **Facts-recovery** ("grandma tier") | Personal facts + secret word (wraps CK; admin theoretically could brute-force with full DB access if facts are weak) | Medium — relies on attacker not having all the facts AND the DB AND a weak secret word. Mitigated by Argon2id. | Movie platform onboarding, low-value codices, casual users |

User can **graduate** from facts-recovery to self-custody at any time: "I'm ready, show me the mnemonic, let me write it down, remove the facts-recovery wrapper." Forward-path is built-in.

### 4.2 Self-custody recovery flow (the elegant one)

Uses the existing Dalos crypto stack — no new primitives needed.

**Setup at signup (or graduation):**
1. User has mnemonic M (24 BIP-39 words derived from initial signup, or chosen by user)
2. Dalos derivation: M → private key sk → CodexPrime account
3. Mnemosyne calls CodexPrime activation via Ouronet Gas Station — registers the account's public key pk on Stoa-chain
4. Server stores: `codex_prime_address` (the ouro address whose pk is on-chain)
5. CK is wrapped under password-derived key. Mnemonic is NOT stored on server (the user writes it down).

**Recovery flow (forgot password, has mnemonic):**
1. User clicks "Recover with mnemonic" in the Mnemosyne SPA
2. UI prompts for 24 words
3. Browser derives sk locally from the mnemonic
4. Server issues a recovery challenge (random nonce)
5. Browser signs the challenge with sk → produces Schnorr signature σ
6. Server reads CodexPrime's pk from Stoa-chain (using the stored `codex_prime_address`)
7. Server verifies σ against pk for the challenge — Schnorr verification on Dalos curve
8. ✅ → server grants password-reset capability
9. Browser derives CK from the mnemonic (separately from sk, e.g. via HKDF(M, "codex-key")) — OR — more cleanly: browser is granted access to download `encrypted_codex_key_facts` if it exists (grandma tier), OR refuses and forces wipe-and-restart for self-custody users who lost the password
10. User sets new password
11. Browser re-wraps CK under the new password-derived key → uploads new `encrypted_codex_key_password`
12. Codex content unchanged

Wait — there's a subtlety. If the user is SELF-CUSTODY (no facts wrapper), and they forgot the password, they need a way to RE-DERIVE CK from the mnemonic. Two designs:

**Design A — CK derived deterministically from mnemonic at signup:**
- CK = HKDF(mnemonic, "mnemosyne-codex-key-v1")
- Server stores `encrypted_codex_key_password` for normal login convenience
- If user loses password but has mnemonic: re-derive CK locally → decrypt codex → set new password
- Drawback: CK is deterministic from mnemonic, so anyone who guesses the mnemonic guesses CK. But guessing a 24-word mnemonic is already ~256 bits of work, so no real degradation.

**Design B — CK is random at signup, stored only under password wrapper:**
- CK = random 256-bit
- `encrypted_codex_key_password` = wrap(CK, KDF(password))
- If user loses password: codex content is gone forever; recovery just verifies identity for "you can create a new codex" but old contents are wiped
- Same model as Mnemonic-only without graceful preservation

**Design A is what we want for content preservation.** Locked.

### 4.3 Why the brain-wallet trap is sidestepped (important context)

Earlier in design discussion we considered deriving CK directly from personal facts ("email + name + DOB + secret word"). This is functionally identical to Bitcoin's failed 2013-2015 **brain wallet** scheme, where attackers drained thousands of BTC by brute-forcing low-entropy passphrase-derived keys.

**The fix in our design**: CK is HIGH-ENTROPY (256-bit random or 24-word-mnemonic-derived = 256 bits). The personal facts are only used as a WRAPPING KEY (to encrypt CK), not as the direct key derivation. An attacker who steals the DB and brute-forces the facts gets ONE user's codex, not all users. And the security still relies on the user's "secret word" being unguessable.

So:
- Facts-recovery is SAFER than brain-wallet because the facts only wrap a random key
- Facts-recovery is still WEAKER than mnemonic-recovery because the facts have lower entropy than 256-bit randomness
- Trade-off is appropriate for grandma-tier (convenience > absolute security); inappropriate for self-custody-tier (absolute security > convenience)

### 4.4 Grandma-tier (facts-recovery) flow

**Setup at signup:**
1. User provides 5-7 personal facts during signup (email, family name, given name, DOB, mother's language, secret word, etc. — exact schema TBD)
2. Facts canonicalized to a stable string (lowercase, trim, sort by field name, etc.)
3. `facts_hash = Argon2id(canonical(facts), salt)` → stored server-side for verification
4. `encrypted_codex_key_facts = encrypt(CK, KDF(canonical(facts)))` → stored server-side
5. Codex still encrypted under random CK (per §3.3)

**Recovery flow:**
1. User clicks "Recover with personal facts"
2. UI prompts for the facts (same schema)
3. Client canonicalizes locally → hashes → sends hash to server (NOT plaintext)
4. Server compares to stored `facts_hash` → match → returns `encrypted_codex_key_facts`
5. Client uses canonical(facts) to derive the wrapping key → unwraps CK
6. Decrypts codex
7. User sets new password
8. Client re-wraps CK under new password → uploads new `encrypted_codex_key_password`
9. Codex content preserved

**Security caveats:**
- If the personal facts schema is too predictable and the secret word is weak, an attacker with database access could brute-force CK. Mitigated by Argon2id at expensive parameters.
- User should be warned at signup: "Don't use your pet's name as the secret word. Don't use your birth year. Don't use anything publicly searchable about you."
- Documented value cap recommendation: "this tier is suitable for codices holding less than $X total value. For higher-value codices, please graduate to mnemonic recovery."

### 4.5 Guardian designation (extending recovery with secondary accounts)

Beyond CodexPrime, a user can designate **Guardians** — additional ouro accounts within their own codex (or external) that can ALSO prove ownership for recovery.

**Constraints (enforced by codex package v0.3.0):**
- Guardian MUST be a Standard Ouronet Account (`isSmart: false`)
- Guardian MUST be activated on Stoa-chain (otherwise no on-chain pubkey to verify against)
- Multiple Guardians allowed per codex (M-of-N not required at v0.1; any single Guardian's signature is sufficient for recovery)
- A Guardian's existence is publicly observable on-chain (the address is in `guardian_addresses` on the Mnemosyne server, but the on-chain account itself is normal)

**Recovery via Guardian:**
- Same flow as §4.2 self-custody recovery, but the user signs the challenge with the Guardian's private key instead of CodexPrime's
- Server verifies against the Guardian's on-chain pubkey
- Useful for users who lost CodexPrime mnemonic but remember a Guardian's seed (e.g., a custom-Dalos-seed account they created themselves)

**Designation flow (UI):**
- User must be authenticated (has codex unlocked)
- Selects an existing ouro account from their codex
- If isSmart === true → refused (Standard accounts only)
- If account not yet activated → warning shown, designation queued
- If activated → designation effective immediately
- Guardian's address added to `guardian_addresses` on server

### 4.6 What the admin (Ancient Holdings) can and cannot do

| Action | Self-custody user | Grandma-tier user |
|---|---|---|
| Decrypt the codex with server-side data alone | ❌ Impossible (no CK access) | ❌ Impossible without brute-forcing facts |
| Initiate a password reset on user's behalf | ❌ Impossible (would need to forge a Dalos Schnorr signature against on-chain pubkey — equivalent to breaking the curve) | ⚠️ Possible IF admin can trigger email reset AND user has weak personal facts |
| Tamper with the codex contents undetectably | ⚠️ Possible (could replace ciphertext with garbage) — but user would notice on next decrypt; mitigated by on-chain blob hash anchoring (v0.2+) | Same |
| Refuse to serve a user (denial of service) | ✓ Possible — but user has their mnemonic, can self-host a Mnemosyne instance, or migrate to another operator | Same caveat |
| Brute-force passwords from `password_verifier` | ⚠️ Theoretically possible but Argon2id makes weak passwords slow; strong passwords infeasible | Same |
| See WHICH codices exist | ✓ Yes (metadata isn't encrypted, only the codex payload) | Same |

The cryptographic guarantees are real. The operational risks (denial of service, observable metadata, weak password brute-force) are standard for the Bitwarden-pattern and accepted as part of the trade-off.

## 5. Onboarding & activation

### 5.1 Three onboarding paths

**Path A — Direct signup at `mnemosyne.ancientholdings.eu`:**
1. User visits the Mnemosyne SPA
2. Reads the explanation of what the codex is and what Mnemosyne does
3. Clicks "Create account"
4. Email + password
5. Optionally: tier choice (self-custody vs grandma-tier)
6. For self-custody: shown the 24-word mnemonic in a "write this down" UX, requires confirmation
7. For grandma-tier: facts schema form, must fill in
8. CodexPrime account is activated on Stoa-chain via the Ouronet Gas Station (paid by Mnemosyne's gas-station budget)
9. Codex generated, encrypted, stored
10. User lands on dashboard

**Path B — Onboarding by another app (e.g. movie platform):**
1. User registers on the movie platform with email + password
2. Movie platform's backend calls Mnemosyne API: `POST /api/onboard { partnerAppId, userEmail, userPassword, ... }`
3. Movie platform's account at the gas station gets debited for CodexPrime activation
4. Mnemosyne mints a codex (random mnemonic, facts-tier by default since user is non-crypto)
5. Returns the user's ouro address to the movie platform
6. Movie platform shows user: "Welcome to Movies. As a side benefit, you also have an ouronet address: Ѻ.xyz. Click here to learn what this means."
7. When user later visits OuronetUI and signs in with same email+password, Mnemosyne serves the same codex

**Path C — Importing existing browser-local codex:**
1. Existing OuronetUI user (browser-local codex) signs up at Mnemosyne
2. SPA reads their browser's localStorage codex
3. Re-encrypts it under Mnemosyne's wrapping
4. Uploads to Mnemosyne
5. CodexPrime is already activated (used the codex before) — no gas needed unless Guardians are designated post-migration
6. Browser-local codex is optionally wiped (with user confirmation)

### 5.2 Activation via Ouronet Gas Station

The existing `STOA_AUTONOMIC_OURONETGASSTATION` account (defined in `@stoachain/ouronet-core/constants`) is the gas station already sponsoring OuronetUI users for chain operations. Extending its scope:

- Mnemosyne registers as a "partner app" with its own budget at the gas station
- Movie platform (and other consumer apps) similarly register as partner apps
- When Mnemosyne triggers a CodexPrime activation on behalf of a user, the gas comes from whichever partner app is sponsoring that user
- Movie platform's signup flow → debits movie platform's gas budget
- Direct signup at mnemosyne.ancientholdings.eu → debits Mnemosyne's own gas budget

**Mechanical implementation (sketch — actual design TBD):**
- Partner app holds an app-key issued by the gas station
- When Mnemosyne triggers activation, it signs the request with the partner-app-key
- Gas station validates the partner-app-key and debits the right budget
- Audit log of "activation X was sponsored by partner Y" stored both on-chain (via the activation tx) and off-chain (audit table)

### 5.3 Why activation matters for v0.1

The Schnorr-recovery flow (§4.2) requires the CodexPrime account to be **activated on-chain** — meaning its public key is published in a Stoa account record. Without activation, there's no on-chain pubkey to verify recovery signatures against. So **activation is mandatory at signup**, not optional.

Cost per activation: a small Pact transaction; gas-station-sponsored. Negligible per user. With a sane budget at the gas station, Mnemosyne can onboard tens of thousands of users without cost concern.

## 6. Consumer integration

### 6.1 The new client package: `@stoachain/mnemosyne-client`

A new npm package, published from a future Mnemosyne monorepo or extracted into stoa-js, providing:

- **HTTP client** for the Mnemosyne API
- **`MnemosyneCloudAdapter`** — implements `CodexAdapter` interface from `@stoachain/ouronet-codex`
- **Auth helpers** (login, signup, password reset, recovery flows)
- **Codex-side encryption helpers** (wrap/unwrap CK with password, with facts, with mnemonic-derived key)

Published independently of the codex package itself (consumers who don't use Mnemosyne don't pull this dep).

### 6.2 Consumer integration example (OuronetUI)

Before (today, OuronetUI v1.2.2):

```typescript
import LocalStorageCodexAdapter from "@/kadena/wallet/LocalStorageCodexAdapter";

<CodexProvider
  adapter={new LocalStorageCodexAdapter("dev")}
  // ... other props
>
```

After (with Mnemosyne integration):

```typescript
import { MnemosyneCloudAdapter } from "@stoachain/mnemosyne-client";

<CodexProvider
  adapter={new MnemosyneCloudAdapter({
    endpoint: "https://api.mnemosyne.ancientholdings.eu",
    sessionToken: userSession.token,
  })}
  // ... other props
>
```

That's the entire integration. The rest of OuronetUI's code (CFM modals, signing, codex hooks) is unchanged.

### 6.3 Backend-side integration (AH hub, movie platform)

Non-React consumers (Node.js servers like the AH hub or movie platform backend) interact via REST API directly. The `@stoachain/mnemosyne-client` package would also export a server-friendly client class for this. Example:

```typescript
// In AH hub or movie platform's backend:
import { MnemosyneServerClient } from "@stoachain/mnemosyne-client/server";

const mnemosyne = new MnemosyneServerClient({
  endpoint: "https://api.mnemosyne.ancientholdings.eu",
  partnerAppKey: process.env.MNEMOSYNE_PARTNER_KEY,
});

// On user signup:
const { ouroAddress } = await mnemosyne.onboardUser({
  email, password, tier: "facts-recovery",
  facts: { /* if facts-recovery */ },
});
```

### 6.4 API surface (preliminary sketch)

```
POST   /api/auth/signup           — create account + kickstart codex
POST   /api/auth/login            — returns session JWT
POST   /api/auth/logout           — invalidates session
GET    /api/codex                 — returns encrypted codex + key wrappers
PUT    /api/codex                 — uploads new encrypted codex (atomic via version counter)
POST   /api/codex/reset-password  — initiates recovery (mnemonic OR facts path)
POST   /api/codex/change-password — change password while authenticated
GET    /api/codex/download        — JSON download for self-custody backup
POST   /api/codex/guardian        — designate / un-designate a Guardian account
POST   /api/onboard               — partner app onboards a user (partner-app-key auth)
GET    /api/user                  — get user account info
DELETE /api/user                  — delete account + wipe codex (irreversible)
```

Endpoint shapes, request/response schemas, error codes — to be detailed in a separate `api-spec.md` once we begin scaffolding.

## 7. Multi-device & multi-consumer behavior

### 7.1 The unified-codex promise

The whole point of Mnemosyne: **one codex per user, accessible from every consumer**. When a user adds an account in OuronetUI on their laptop, that account appears in OuronetUI on their phone, in the AH admin UI (if they're an admin), in the movie platform's wallet view (if it has one), etc. Single source of truth.

### 7.2 Conflict resolution

When two consumers (or two instances) write to the same codex simultaneously, we need a conflict policy. Three options:

| Policy | Pros | Cons | Verdict for v0.1 |
|---|---|---|---|
| **Last-writer-wins (LWW)** | Simplest; matches user mental model 80% of the time | Can silently overwrite the other write | ⚠️ Acceptable if uncommon |
| **Optimistic concurrency (version counter)** | Detects conflicts; client retries on stale write | Requires UX for "conflict — your changes weren't saved, please retry" | ✓ Recommended for v0.1 |
| **Vector clocks / CRDTs** | Auto-merges without user-visible conflicts | Complex implementation; harder to reason about | Too heavy for v0.1; consider for v0.3+ |

**Recommended v0.1 policy:** optimistic concurrency. Server tracks a monotonically increasing `version` per codex. Client includes `if-version: N` header on PUT. If server's current version is > N, returns 409 Conflict. Client must re-fetch, re-apply local changes, retry.

This means clients should batch their writes (debounce small changes; PUT a complete codex blob at a sensible interval, not per-key-stroke).

### 7.3 Per-consumer settings namespacing

Today's UiSettings has OuronetUI-specific fields mixed with generic codex settings. With multiple consumers writing to the same codex, this becomes a collision risk.

**v0.3.0 codex package change (see §9):** introduce `consumerSettings: Record<string, IConsumerSettings>` registry, namespaced by consumer name. OuronetUI writes to `consumerSettings["OuronetUI"]`. AH hub to `consumerSettings["AncientHoldings"]`. Movie platform to `consumerSettings["MoviePlatform"]`. No collisions.

### 7.4 What lives in the codex vs what stays app-local

A clean rule, drawn here while it's clean:

| Category | Lives in codex | Lives in app |
|---|---|---|
| Identity-adjacent (keys, addresses, settings) | ✓ | |
| Cross-consumer-useful (address book, watch list) | ✓ | |
| Auth/signing preferences | ✓ | |
| Personalization (theme, language, units) | ✓ | |
| Per-consumer UI preferences (e.g. selectedNode for OuronetUI) | ✓ (namespaced under consumerSettings) | |
| Application data (movie history, pool position cache) | | ✓ |
| Large or churny state (transaction history cache, balances) | | ✓ |
| Anything that doesn't make sense in another consumer | | ✓ |

Codex stays meaningful and reasonably small. App-local state stays app-local.

## 8. Decentralization layers (deferred)

The v0.1 core is centralized (Mnemosyne server is the source of truth for codex data). Decentralization is layered on top in subsequent releases as **optional enhancements**, not core architecture rewrites:

### 8.1 v0.2 — Arweave snapshot layer (L2 pattern)

- Mnemosyne server batches a snapshot of the entire encrypted vault state and writes it to Arweave on a schedule (e.g., hourly)
- Each snapshot is signed by Mnemosyne's server key and timestamped
- If Mnemosyne server dies, anyone with the Arweave reading capability can recover the vault (just an encrypted blob anyone can fetch from `arweave.net/<tx-id>`)
- Cost at 1000 users: 1 tx/hour × ~$1/tx = ~$24/day = ~$720/month, acceptable for a paid service tier
- Cost can be drastically reduced by snapshot diffing (only post changes, not full state)

This is the "L2-on-Arweave" pattern the user proposed during design discussion. It's a server with periodic on-chain commits — same idea as Optimistic / ZK rollups for Ethereum, but for codex data instead of token state.

### 8.2 v0.3 — Per-codex Arweave pinning (user-funded)

- Free tier: codex lives only on Mnemosyne server + in periodic vault snapshots
- Paid tier: user pays per codex update to have their specific codex pinned individually to Arweave
- Useful for high-value codex holders who want individual on-Arweave durability rather than relying on Mnemosyne's snapshot cadence
- Pricing: ~$0.50-1 per update; user controls when to upgrade

### 8.3 v1.0 — On-chain Stoa pointers (immutable Pact module)

- A Pact module `mnemosyne.vault` deployed on Stoa-chain with **immutable governance** (governance keyset initialized with an unsatisfiable predicate — see design discussion for the trick)
- Stores per-user pointer: `{ ouro-addr → arweave-tx-id of latest codex blob }`
- Module's `write-codex` enforces guard from the user's account-guard (only the user can update their own pointer)
- Module's `read-codex-pointer` is permissionless (anyone can verify ownership)
- This makes ownership cryptographically enforceable on-chain, not just trusted to the Mnemosyne server
- Any third-party Mnemosyne-compatible client can use the same on-chain pointers — opens the door to a true ecosystem of Mnemosyne-pattern implementations
- Cost per codex update: small Stoa tx (gas-station-sponsored, same as activation)

### 8.4 SPA permaweb deploy (independent of the above)

- Mnemosyne SPA deployed to Arweave per major release (~$10-30 per deploy)
- Users can access the SPA at `arweave.net/<tx-id>` as a censorship-resistant fallback
- Same pattern applies to OuronetUI's SPA — monthly permaweb deploys make OuronetUI itself censorship-resistant at the frontend level

### 8.5 What "decentralized" actually means after these layers

- **v0.1**: centralized SaaS, Mnemosyne server is single point of trust + failure
- **v0.2**: server can die without data loss (Arweave snapshots are recoverable by anyone)
- **v0.3**: per-codex individual Arweave durability available as opt-in
- **v1.0**: ownership is cryptographically verifiable on Stoa-chain; third-party clients can interop
- **At v1.0**: Mnemosyne is *operationally* decentralized — anyone can run a Mnemosyne node, anyone can read/verify codices independently of Ancient Holdings. AH-as-an-operator becomes "the first / largest operator," not the only one.

This is the Bitcoin/Coinbase pattern applied to codex management. Bitcoin (decentralized network) plus Coinbase (centralized custodian for non-crypto users). Both layers coexist; neither depends on the other for survival.

#### 9.1.4 Consumer settings registry

Replaces the loosely-typed `UiSettings[extra: string]: unknown` escape hatch with a properly namespaced registry.

New types:

```typescript
export interface IConsumerSettings {
  /** Canonical identifier of the consumer (e.g. "OuronetUI",
   *  "AncientHoldings", "MoviePlatform", "Mnemosyne"). */
  consumerName: string;

  /** Semver version of the consumer that last wrote these settings.
   *  Used for migrations when the consumer's settings schema evolves. */
  consumerVersion: string;

  /** Schema version internal to THIS consumer's settings shape.
   *  Each consumer evolves independently. */
  schemaVersion: number;

  /** Opaque-to-other-consumers settings blob, typed by the consumer.
   *  Other consumers should preserve this verbatim on save (additive
   *  convention — don't drop fields you don't recognize). */
  settings: Record<string, unknown>;

  /** ISO timestamp of last write. */
  lastUpdatedAt: string;
}

export interface CodexSnapshot {
  // ... existing fields ...

  /** Per-consumer settings registry (v0.3.0+). Keyed by consumerName.
   *  Each consumer reads/writes ONLY its own entry; other entries are
   *  preserved verbatim. This replaces ad-hoc keys-on-UiSettings. */
  consumerSettings?: Record<string, IConsumerSettings>;
}
```

New hook:

```typescript
export function useConsumerSettings<T>(opts: {
  consumerName: string;
  schemaVersion: number;
  defaults: T;
  migrations?: Migration[];
}): {
  settings: T;
  updateSettings: (patch: Partial<T>) => Promise<void>;
  reset: () => Promise<void>;
};
```

The package handles:
- Namespacing (writes go to `consumerSettings[consumerName]`)
- Migration application (if loaded schema_version < current, runs migrations)
- Defaults (if no settings exist yet for this consumer, returns `defaults`)
- Preservation (other consumers' entries are untouched on save)

#### 9.1.5 Schema migration framework

Currently `schemaVersion` is a number but there's no formal migration mechanism. Formalize it.

```typescript
export interface SchemaMigration {
  fromVersion: number;
  toVersion: number;
  migrate: (snapshot: CodexSnapshot) => CodexSnapshot;
  description: string;
}

// Registered in the package, applied on adapter.init()
export const SCHEMA_MIGRATIONS: SchemaMigration[] = [
  // ... migrations added per release ...
];

// New action:
export interface CodexStoreActions {
  /** Migrate a loaded snapshot to the current schema version.
   *  Called automatically by init() if needed. Idempotent. */

// New helper:
export function canConsumerWrite(currentSchemaVersion: number): boolean;
```

`canConsumerWrite` returns false if the loaded codex is at a NEWER schema than this package version understands — protects against an outdated client overwriting fields it doesn't know about.

## 10. Open questions

These need answers before scaffolding begins. Many are tactical implementation choices that don't change the design but do affect the implementation plan.

### 10.1 Technical

1. **Backend framework.** Next.js (matches AH hub) / Express / Hono / Fastify — pick one based on team familiarity + ecosystem.
2. **Database engine.** SQLite (single-server, matches AH hub) vs Postgres (scales horizontally if needed later). SQLite is simpler at v0.1; Postgres avoids a migration if we grow.
3. **Auth library.** Roll our own with Argon2id (~1 week, fewer deps) vs Lucia (lightweight, modern) vs Auth.js (more features, more bloat) vs Supabase (fastest to ship, vendor lock-in). Need to evaluate.
4. **DB encryption at rest.** Server-side disk encryption (SQLCipher / Postgres TDE) gives defense-in-depth but doesn't help against a compromised process. Discuss.
5. **Session token strategy.** JWT (stateless, no DB lookup per request) vs DB-backed sessions (revokable, can be invalidated server-side). JWT is simpler; revocation is harder.
6. **Multi-device sync conflict resolution.** Optimistic concurrency (recommended in §7.2) is the v0.1 answer. UX for the rare conflict case TBD.
7. **API rate limiting strategy.** Per-user / per-IP / both. Prevents brute-force on the recovery facts endpoint specifically.
8. **2FA option.** Optional TOTP-based 2FA for additional account security. Probably v0.2+; not required for v0.1.

### 10.2 Product

1. **Pricing model.** Free for grandmas (via partner-app gas station)? Paid Arweave-permanence tier? Bundled with Mnemosyne API access for partner apps? Affects business model + may inform regulatory posture.
2. **Movie-platform integration spec details.** Once we know the movie platform's tech stack, design the partner-app onboarding API in detail. For now, sketched in §6.3.
3. **Brand / messaging.** "Mnemosyne" vs alternative names; landing-page copy; explainer animations. Out of scope for engineering, but worth flagging.
4. **Self-hosting story.** Should we publish the Mnemosyne backend as deployable software so paranoid users can run their own? Affects open-source license choice + dependency choices.
5. **Geographic deployment.** EU-only initially? Multi-region from launch? Affects CDN + database choice.

### 10.3 Strategic

1. **Codex package v0.3.0 scope confirmation.** Anything in §9 we want to defer / add?
2. **Mnemosyne in the cross-pollinate workspace.** When do we register it in `.wasp/cross-pollinate.yml`? After v0.1 scaffold lands? After v0.1 ships?
3. **Sequencing OuronetUI migration to Mnemosyne.** When does OuronetUI gain a "Sync with Mnemosyne" opt-in flow? Voluntary at first, mandatory at some future point?
4. **AH hub integration.** AH hub today encrypts its codex under SECRETS_MASTER_KEY (env). Does it migrate to Mnemosyne? Probably not — the hub is a single-codex system with autonomous signing; Mnemosyne is multi-user. But we should think about whether the hub's codex pattern *should* be served by Mnemosyne for resilience (Mnemosyne stores the hub's encrypted codex blob; hub still has the master key client-side to unwrap).

## 12. Risks & deferred topics

### 12.1 Regulatory posture

The owner's stated vision is **jurisdictionless** ("system that works and helps consumers achieve their goal... no entity should dictate how it can be used"). The technical design respects this — the cryptographic guarantees make Mnemosyne **technically non-custodial** at the self-custody tier. But the *operator* (Ancient Holdings) still has a legal posture to think about for the SaaS layer, especially for grandma-tier users where the operator's role is more involved.

**Plan:** discussed in design conversation but DEFERRED. Approach when scoping v0.1 launch:
- For self-custody users: the operator demonstrably cannot access user keys. This is the strongest defensible legal position. Document it clearly in ToS.
- For grandma-tier users: the operator has theoretical recovery assistance capability. This is custodial-flavored. Probably needs MiCA CASP consideration if operated from the EU.
- Long-term: convert Ancient Holdings to a foundation (already planned per owner's note) to take the entity-as-operator dimension off the table.

Owner explicitly does NOT want this to delay technical work. Counsel engagement is parallel to engineering.

### 12.2 Brain-wallet trap

We considered (and rejected) deriving the codex key directly from personal facts. Bitcoin's history with brain wallets is conclusive — direct human-input-derived keys are catastrophically vulnerable. Our design uses facts only as a WRAPPING factor around a random key. This is meaningfully safer but still weaker than mnemonic-based recovery. Clearly documented in §4.3-4.4.

### 12.3 Multi-implementation governance

Not relevant at v0.1 (single implementation, Ancient Holdings operates). Becomes relevant if at v1.0+ we open up the protocol for other operators. Punt until then.

### 12.4 Operator key compromise

If Mnemosyne's server keys (TLS, JWT signing, partner-app validation) are compromised, the attacker can issue valid sessions and trigger Arweave snapshots with fake data. Mitigations:
- Sensitive operations (Arweave snapshot signing, partner-app validation) use HSM-stored keys
- Audit log to detect anomalous activity
- Out-of-band notification to users for high-risk operations (password resets, Guardian designations, etc.)

These are operational concerns, not architectural. Standard SaaS security hygiene.

### 12.5 Loss of mnemonic AND password

For self-custody users: codex is gone. Documented at signup with strong UX.
For grandma-tier users: facts-recovery is the safety net. If facts are also lost: codex is gone.

This is the inherent trade-off of the design. Cannot be solved without making the operator custodial (which the owner explicitly opposes).

## 13. Glossary

| Term | Meaning |
|---|---|
| **Codex** | An encrypted multi-wallet store containing kadena seeds, ouro accounts, pure keypairs, address book, watch list, settings. Source-of-truth for a user's ouronet identity. Defined by `@stoachain/ouronet-codex`. |
| **CodexPrime** | The first ouro account in a codex, derived from the same mnemonic as the Prime Codex Seed. Structurally unremovable. The canonical on-chain identity of the codex. |
| **Prime Codex Seed** | The kadena seed that birthed the codex (the original 12-or-24-word mnemonic). Structurally unremovable. |
| **Guardian** | A non-CodexPrime ouro account designated as an additional recovery factor. New in codex package v0.3.0. |
| **CK** (Codex Key) | The random 256-bit AES key that encrypts a codex blob in Mnemosyne. Wrapped by password and/or facts. |
| **Mnemosyne** | This project. Centralized SaaS for codex storage with cryptographic non-custodial guarantees. |
| **SaaS** | Software as a Service. Cloud-hosted software accessed via browser/API. |
| **SPA** | Single Page Application. JavaScript app that loads once and re-renders dynamically. |
| **Self-custody tier** | Recovery via mnemonic only. Admin cannot help. Strong guarantee. |
| **Grandma tier** | Recovery via personal facts wrapping the random codex key. Admin has weaker (still cryptographically bounded) recovery capability. Easier UX. |
| **Schnorr signature** | The signature scheme on the Dalos elliptic curve. Used for the on-chain-pubkey recovery verification. |
| **Ouronet Gas Station** | The existing `STOA_AUTONOMIC_OURONETGASSTATION` account that sponsors gas for OuronetUI users. Extended scope for Mnemosyne: sponsors CodexPrime activations during signup. |
| **Partner app** | A consumer service (movie platform, etc.) that onboards users through Mnemosyne. Has its own gas-station budget for sponsoring activations. |
| **Permaweb** | Arweave's "permanent web hosting" — static SPAs deployed once, stay online indefinitely via Arweave's endowment-funded storage model. |
| **L2 / Layer 2** | "Layer 2" — an off-chain or off-decentralized-storage system that periodically commits state to a more decentralized substrate. Mnemosyne's Arweave snapshot model is L2-on-Arweave. |
| **Bitwarden pattern** | Centralized SaaS + client-side encryption + multi-device sync. Server stores opaque ciphertext; cannot decrypt. The architectural template Mnemosyne follows. |
