Ouronet Accounts

Anatomy of an Ouronet Account вҖ” the Public Key, Payment Key, and Guard, plus the Smart-account-only Governor and Sovereign slots. The architecture behind the Сә. and ОЈ. address prefixes.

Two flavours, one architecture

An Ouronet Account is built on the Dalos Elliptic Curve and lives above the StoaChain native account layer. Every Ouronet Account has a 160-glyph Dalos public key that does not change for the account's lifetime вҖ” that's the immutable identifier you see as Сә.xxx or ОЈ.xxx.

Around that immutable public key, an account composes a small set of rotatable slots. Each slot points at on-chain entities (keys, accounts, guards) that the account uses to authorize behaviour. Slots can be rotated independently as your security posture changes.

Сә. — Standard Account

The simpler flavour. Has Public Key, Payment Key, and Guard. Designed for everyday usage вҖ” single-account, single-flow operations.

ОЈ. — Smart Account

Standard + two extra slots: Governor (for non-key-based authorization patterns) and Sovereign (a Standard Account that has sovereignty over this Smart Account). Designed for governance, multi-party authorization, and protocol-level integrations.

The slots

Public Key immutable

The Dalos 160-glyph public key derived from the account's seed at creation. Defines the account's identity. Never changes вҖ” this is what Сә.xxx / ОЈ.xxx visually encodes.

Stored on chain immutably. If the codex's local copy and the chain disagree, the chain wins. Mnemosyne's UI shows a "codex matches chain" check next to the public-key panel to surface drift.

Payment Key rotatable вҖ” STOA

The StoaChain-native account that pays gas for any operation involving this Ouronet Account. At creation this is a k: account вҖ” a simple keyset-backed account. It can be rotated to any StoaChain account type:

  • вҖў k: вҖ” single-key keyset (the default at creation)
  • вҖў w: вҖ” multi-key keyset (multi-sig: 2-of-2, 3-of-5, any-of-N, all-of-N)
  • вҖў c: вҖ” capability-guarded (autonomous accounts whose payment is authorized by Pact module logic, not a private key)
  • вҖў u: вҖ” user-guarded (custom Pact predicate functions)
  • вҖў r: вҖ” keyset-reference (points to a named on-chain keyset)
  • вҖў Any named StoaChain account

Rotating the Payment Key changes who pays and what authorizes the payment, without changing the Ouronet Account's identity. Common rotation: keyset вҶ’ capability-guard, to make the account fully autonomous under a Pact module.

Guard rotatable

At creation: a single-key keyset (one Stoa key, keys-all predicate), making the account behave as a normal single-signature wallet. The Guard slot is technically any Pact guard вҖ” Pact recognises four guard types, and the slot accepts all of them:

  • вҖў keyset вҖ” defined inline at transaction-upload time (e.g. { keys: [...], pred: keys-any })
  • вҖў keyset-ref вҖ” a reference to a keyset already defined on chain in some namespace (e.g. ouronet-ns.dh_sc_dalos-keyset); the keys can be updated separately from the account
  • вҖў user-guard вҖ” any Pact function returning bool can be wrapped as a guard; whatever logic the function enforces becomes the authorization rule
  • вҖў capability-guard вҖ” a Pact capability can itself be wrapped as a guard; when that capability is composed in a transaction, the guard is satisfied

That said, the Guard slot is mentally designed for the first two вҖ” keysets and keyset-refs. It's the "who has the keys" slot. For the more programmable guard types (user-guards and capability-guards), Smart accounts give you a separate slot вҖ” the Governor вҖ” to keep the authorization structure easy to read at a glance.

Governor Smart only rotatable

Smart-account-only. Another guard slot, mentally designated for the programmable guard types вҖ” user-guards and capability-guards. Where you put authorization logic that says "this account is authorized when these on-chain conditions hold" rather than "this account is authorized when these specific keys sign."

Common patterns:

  • вҖў user-guard composing multiple capability-guards (any-of, all-of)
  • вҖў capability-guard tied to a specific Pact-module capability (e.g. REMOTE-GOV)
  • вҖў Module-defined autonomous logic

Technically Pact would accept the same content in the Guard slot вҖ” nothing about the chain stops you from putting a complex user-guard there. The Guard / Governor separation is a human-readability choice (see the design-rationale sidebar below).

Sovereign Smart only rotatable

Smart-account-only. Points to a Standard Ouronet Account (Сә.xxx) that has sovereignty over this Smart Account.

Satisfying the Sovereign account's ownership rules = satisfying the Smart Account's sovereignty. In practice: if you can sign on behalf of the Sovereign, you can authorize sovereign operations on the Smart Account too.

Why exist: it gives a clean "override path" for Smart accounts. The day-to-day authorization runs through Guard + Governor; deep / structural changes (rotations, mode changes, module upgrades) route through the Sovereign as a human-controllable escape hatch.

Why three slots instead of one вҖ” the design rationale

Technically, a single guard slot would have been enough. Pact lets you compose arbitrarily complex guards вҖ” you could put a single user-guard in the Guard slot that internally composes a keyset, capability checks, sovereign overrides, and any other logic you need. One slot, infinite expressiveness.

The Guard / Governor / Sovereign separation exists because the system was designed for humans, not for machines. A machine doesn't care whether your authorization rules live in one compound guard or three discrete slots вҖ” it just runs the predicates. A human reading an account's structure needs to see, at a glance, what authorizes what. The three slots answer three distinct questions:

  • Guard вҖ” “Who has the keys?” A keyset or keyset-ref. The "wallet" semantic вҖ” direct private-key-based control.
  • Governor вҖ” “What programmable conditions authorize this?” A user-guard or capability-guard. The "logic" semantic вҖ” autonomous, contract-driven, governance-pattern authorization.
  • Sovereign вҖ” “Whose human override sits above this?” A Standard Ouronet Account. The "escape hatch" semantic вҖ” when something at the Guard or Governor layer needs structural change.

You're free to ignore the mental separation entirely. Pact treats all guard types as just "a guard" вҖ” you can put a capability-guard in the Guard slot, a single-key keyset in the Governor slot, anything you want. The slots are conventions to help you build accounts whose ownership structure is readable, not constraints the chain enforces.

The trade-off accepted: more slots = slightly more setup complexity at account creation, in exchange for an ownership structure a human can understand without unwrapping a deeply-nested compound guard.

The Governor accepts authorization from any of five different Pact-module capabilities. The Public Key is the only thing that stayed constant from creation.

Why this matters for Codex Consumers

The slot model makes Ouronet Accounts much more flexible than typical single-key blockchain accounts. A Codex Consumer (OuronetUI, AncientHoldings, the upcoming Demiourgos Streaming Platform) can build features like:

  • вҖў Time-locked sovereign override (Sovereign signs a delayed rotation)
  • вҖў Capability-guarded autonomous payments (no key, just protocol logic)
  • вҖў Multi-seed backup keys without rotating the public identity
  • вҖў Module-driven governance of accounts (DAO-style)
  • вҖў Account types specialised per purpose (treasury, payments, governance, sovereign control)

None of these require changing the account's Public Key. Anyone resolving ОЈ.xxx always finds the same identity вҖ” what's behind it can change as the user's needs evolve.
